# 拍豆场景与会员权益统一配置设计

**日期**：2026-07-13
**状态**：已确认
**范围**：管理后台、业务后端、小程序

## 1. 背景

当前系统存在三个真实拍豆扣费场景：发布约拍、发起约拍申请、查看联系方式。三个场景的基础消耗均写死为后端常量，只有发布约拍接入了会员报价。管理后台“权益配置”页面只能编辑一条发布约拍会员减免规则，并直接展示 `BASIC`、`SUBMIT_YPAT`、`PPD_DISCOUNT` 等英文编码。

小程序发布约拍页面已能加载会员报价，但费用卡片位于表单中部，与固定提交按钮分离。发起约拍申请页面仅展示固定消耗，没有会员报价。页面展示、余额判断和后端实际扣费因此没有统一的数据来源。

本设计将三个场景的基础费用与会员优惠统一纳入“会员系统 → 权益配置”，并让所有扣费入口在服务端事务内使用同一报价结果。

## 2. 目标

1. 后台可分别配置发布约拍、发起约拍申请、查看联系方式的基础消耗拍豆。
2. 后台可按会员等级和使用场景配置优惠拍豆、最低实扣、生效状态、启用状态与说明。
3. 权益配置页面只展示中文业务名称，不直接展示英文编码。
4. 发布约拍、发起约拍申请、查看联系方式均展示真实报价。
5. 后端余额校验、流水记录与实际扣费使用事务内重新计算的报价，不信任前端金额。
6. 发布页与报名页采用已确认的固定底栏设计：费用摘要与提交按钮同一行，按钮占可用宽度的 45%。
7. 报名页的关键拍豆数值使用主题色，不再使用橙色或普通正文色。

## 3. 非目标

1. 不调整会员套餐购买、支付或有效期逻辑。
2. 不改变保证金配置页面与保证金接口。
3. 不支持运营自由创建未知场景或未知权益类型。
4. 不在本次设计中增加百分比折扣；会员优惠继续使用固定减免拍豆。
5. 不把客户端显示金额作为服务端扣费依据。

## 4. 术语与中文映射

英文编码只用于接口、数据库和服务端分支。后台筛选器、表格、编辑弹窗和小程序均展示中文名称。

### 4.1 使用场景

| 内部编码 | 中文名称 | 当前扣费入口 |
|---|---|---|
| `SUBMIT_YPAT` | 发布约拍 | `YpatInfoService`(约拍信息服务) |
| `APPLY_YPAT` | 发起约拍申请 | 约拍申请与作品快捷申请 |
| `VIEW_CONTACT` | 查看联系方式 | 联系方式解锁服务 |

约拍详情发起申请与作品详情快捷申请共用 `APPLY_YPAT`，保证同一种用户行为采用同一费用规则。

### 4.2 会员等级

| 内部编码 | 中文名称 |
|---|---|
| `BASIC` | 基础会员 |
| `PLUS` | 高级会员 |
| `PRO` | 专业会员 |

后台不允许权益规则保存未知会员等级。若历史数据存在未知编码，接口返回“其他会员等级”，同时标记为不可编辑并记录告警，不把原始英文编码显示给运营人员。

### 4.3 权益类型

| 内部编码 | 中文名称 |
|---|---|
| `PPD_DISCOUNT` | 拍豆减免 |

## 5. 方案选择

### 5.1 采用方案：基础费用与会员规则分表，后台统一呈现

基础费用是所有用户的业务定价，会员减免是基于会员等级的权益。两者生命周期不同，因此分别保存：

- 新增拍豆场景配置表，保存每个场景的基础消耗。
- 扩展现有会员权益规则表，为三个场景保存各会员等级的减免规则。
- 后台通过统一聚合接口读取和保存，运营人员只看到一个“权益配置”页面。

该方案避免把基础费用伪装成某个会员等级的规则，也避免在两个后台页面重复维护同一业务。

### 5.2 未采用方案

1. **单表混合保存**：在会员规则表中增加特殊“非会员”等级保存基础费用。数据含义混乱，容易被会员状态判断误用。
2. **新增独立拍豆配置菜单**：基础费用和会员优惠分散在两个入口，运营修改一个场景时难以确认最终实扣。
3. **继续使用后端常量**：无法满足不同场景可配置，也无法保证后台展示与实际扣费一致。

## 6. 数据模型

### 6.1 新增拍豆场景配置表

表名：`t_ppd_scene_config`(拍豆场景配置表)

| 字段 | 类型 | 约束 | 说明 |
|---|---|---|---|
| `id` | `BIGINT` | 主键 | 配置编号 |
| `scene` | `VARCHAR(32)` | 唯一、非空 | 使用场景编码 |
| `original_ppd` | `INT` | 非空、非负 | 基础消耗拍豆 |
| `description` | `VARCHAR(256)` | 可空 | 中文说明 |
| `updated_at` | `DATETIME` | 非空 | 更新时间 |

基础费用配置必须存在，不提供停用状态。需要免费时将基础消耗配置为 0，避免“停用后按什么价格扣费”的歧义。

### 6.2 扩展现有会员权益规则

继续使用 `t_member_benefit_rule`(会员权益规则表)，唯一键保持“会员等级 + 场景 + 权益类型”。三个场景均使用 `PPD_DISCOUNT`(拍豆减免)。

现有字段含义保持不变：

- `discount_ppd`：固定减免拍豆。
- `min_actual_ppd`：最低实扣，不能高于当前场景基础费用。
- `effective`：业务链路是否已接入该规则。
- `status`：运营是否启用该规则。
- `description`：中文说明。

### 6.3 初始数据

三个场景的基础费用均以当前生产常量 3 拍豆初始化，保证迁移前后非会员扣费不变。

现有发布约拍会员规则原样保留。发起约拍申请与查看联系方式新增三种会员等级规则，初始优惠为 0、最低实扣为 0、已接入、已启用。运营配置优惠前，会员实扣与非会员一致。

## 7. 报价规则

统一报价服务接收用户编号和场景编码，返回：

```text
场景
会员是否有效
会员等级编码与中文名称
基础消耗拍豆
会员优惠拍豆
实际扣除拍豆
规则是否生效
```

计算顺序：

1. 从拍豆场景配置表读取基础消耗。
2. 查询用户当前有效会员等级。
3. 查询“会员等级 + 场景 + 拍豆减免”的权益规则。
4. 非会员、会员过期、规则未生效或规则停用时，优惠为 0。
5. 有效优惠为 `min(配置优惠, 基础消耗)`。
6. 实扣为 `max(最低实扣, 基础消耗 - 有效优惠)`。
7. 最终实扣不得小于 0，也不得高于基础消耗。

示例：基础消耗 3、会员优惠 2、最低实扣 0，则实扣 1。

## 8. 接口设计

### 8.1 小程序报价接口

扩展现有接口：

```text
GET /member/benefit/quote?scene={scene}
```

允许场景：`SUBMIT_YPAT`、`APPLY_YPAT`、`VIEW_CONTACT`。其他值返回参数错误，不按 0 拍豆处理。

返回对象增加中文名称字段，客户端无需自行散落映射：

```json
{
  "scene": "APPLY_YPAT",
  "sceneName": "发起约拍申请",
  "memberActive": true,
  "levelCode": "BASIC",
  "levelName": "基础会员",
  "originalPpd": 3,
  "discountPpd": 2,
  "actualPpd": 1,
  "ruleEffective": true
}
```

### 8.2 后台统一配置接口

```text
GET /admin/member/benefit-configs
PUT /admin/member/benefit-configs/{scene}
```

查询接口一次返回三个场景及其会员规则。保存接口在一个事务中更新当前场景基础费用和全部会员等级规则。

保存校验：

1. 场景必须在白名单中。
2. 基础消耗、优惠拍豆、最低实扣必须为非负整数。
3. 最低实扣不能高于基础消耗。
4. 优惠拍豆可以高于基础消耗，但报价时最多减到最低实扣；后台同时给出风险提示。
5. 会员等级和权益类型必须在白名单中。
6. 更新时校验版本或更新时间，避免两个管理员互相覆盖。

现有会员规则查询和单条保存接口暂时保留，供兼容测试与回滚使用；新版后台只调用统一配置接口。

## 9. 后端扣费改造

三个扣费入口必须在各自事务内重新调用报价服务：

1. 发布约拍：替换固定发布消耗，使用 `SUBMIT_YPAT` 报价。
2. 发起约拍申请：约拍申请与作品快捷申请都使用 `APPLY_YPAT` 报价。
3. 查看联系方式：使用 `VIEW_CONTACT` 报价。

每个入口按以下顺序执行：

1. 校验业务是否允许操作，例如重复申请、重复解锁。
2. 读取事务内最新报价。
3. 校验最新拍豆余额是否大于等于实扣。
4. 扣减实扣拍豆。
5. 拍豆流水记录实际扣减值、场景编码和中文说明。
6. 完成业务写入。

客户端传入的基础费用、优惠或实扣字段一律忽略。

## 10. 管理后台设计

路由保持 `/member/rule`，菜单名称保持“权益配置”，不新增保证金系统菜单。

### 10.1 页面结构

采用已确认的“按场景分栏”方案：

```text
┌ 发布约拍 ┬ 发起约拍申请 ┬ 查看联系方式 ┐
└──────────────────────────────────────┘
  基础消耗拍豆  [ 3 ] 拍豆                 [保存]
  非会员及无有效优惠时按此扣除

  会员等级 | 优惠拍豆 | 最低实扣 | 生效 | 状态 | 说明 | 操作
  基础会员 |    2     |    0     | 是   | 启用 | ...  | 编辑
  高级会员 |    2     |    0     | 是   | 启用 | ...  | 编辑
  专业会员 |    3     |    0     | 是   | 启用 | ...  | 编辑
```

页面不使用分页。每个场景最多展示当前支持的会员等级，切换场景后只显示当前场景规则。

### 10.2 中文展示规则

- 等级：基础会员、高级会员、专业会员。
- 场景：发布约拍、发起约拍申请、查看联系方式。
- 权益类型：拍豆减免。
- 生效：已生效、未生效。
- 状态：启用、停用。
- 表格、筛选器、编辑弹窗均不得直接展示内部编码。

编辑弹窗中的等级、场景、权益类型使用只读中文文本。优惠拍豆、最低实扣使用数字步进器；生效和状态使用开关；说明使用多行文本框。

### 10.3 保存反馈

- 保存期间禁用当前场景编辑与保存按钮。
- 保存成功后重新读取当前场景，显示“配置已保存”。
- 版本冲突时提示“配置已被其他管理员修改，请刷新后重试”。
- 校验错误直接定位到对应字段，不关闭编辑弹窗。

## 11. 小程序设计

### 11.1 发布约拍页面

删除表单中部的会员权益卡片。固定底栏改为左右同一行：

- 左侧费用摘要占 55%。
- 右侧“发布约拍”按钮占 45%。
- 第一层突出“本次实扣 N 拍豆”，金额使用主题色。
- 第二层展示“原价 N”和“会员优惠 -N”。原价仅在存在优惠时使用删除线。
- 无有效优惠时展示“暂无会员优惠”，实扣等于基础消耗。

页面加载 `SUBMIT_YPAT` 报价。报价未加载完成时显示加载状态并禁用发布按钮；报价失败时显示“费用加载失败，点击重试”，不允许用户在未知价格下提交。

### 11.2 发起约拍申请页面

复用已确认的两层费用摘要：

- 左侧 55%，右侧“确认提交”按钮 45%。
- 第一层展示主题色实扣金额。
- 第二层展示余额、原价和会员优惠。
- `bottom-bar__value`(底栏数值)统一使用主题色。
- 余额不足判断、充值提示所需金额均使用报价中的实际扣除，不再使用固定常量。

页面加载 `APPLY_YPAT` 报价。提交前刷新用户余额，后端仍在事务内重新报价。

### 11.3 查看联系方式

确认弹窗展示：

```text
本次实扣 1 拍豆
原价 3 拍豆，基础会员优惠 2 拍豆
```

无有效优惠时展示“本次消耗 3 拍豆，暂无会员优惠”。已解锁联系方式时不再显示扣费确认，也不重复扣费。

### 11.4 响应式与安全区

- 固定底栏继续包含底部安全区。
- 费用摘要设置最小宽度为 0，次要文本允许省略，关键实扣金额和按钮文案不得截断。
- 提交按钮保持现有高度，宽度使用 45%，不使用固定像素宽度。
- 页面正文增加与底栏实际高度一致的底部占位，最后一个表单项不能被遮挡。

## 12. 前端状态管理

将会员状态仓库中仅支持发布约拍的单一报价改为按场景缓存：

```text
quotes[SUBMIT_YPAT]
quotes[APPLY_YPAT]
quotes[VIEW_CONTACT]
```

提供 `refreshBenefitQuote(scene)`(刷新权益报价) 方法。页面只读取自身场景，避免一个页面刷新覆盖另一个场景报价。

接口与前端类型将场景限制为三个允许值，避免任意字符串绕过类型检查。

## 13. 异常与一致性

1. 客户端报价失败：禁用会扣费的主要操作并提供重试。
2. 后台配置缺失：迁移阶段服务端临时回退当前常量 3，并记录错误日志；完成迁移验证后，配置缺失视为服务异常。
3. 会员规则缺失：按无会员优惠处理，不影响基础业务。
4. 余额在报价后发生变化：以后端事务内余额与报价为准，余额不足返回统一错误。
5. 后台保存失败：事务回滚基础费用与会员规则，不能只保存一半。
6. 优惠大于原价：实际优惠受最低实扣与 0 下限约束。

## 14. 数据迁移与发布顺序

1. 创建 `t_ppd_scene_config`，写入三个场景的 3 拍豆基础配置。
2. 为发起约拍申请和查看联系方式补齐会员规则，初始优惠为 0。
3. 上线统一报价服务与后台聚合接口。
4. 改造三个后端扣费入口，并完成服务测试。
5. 上线后台权益配置页面。
6. 上线小程序发布页、报名页和查看联系方式报价展示。
7. 验证拍豆余额、流水与业务结果一致后，移除业务服务对固定拍豆常量的依赖。

## 15. 测试策略

### 15.1 后端

- 三个场景分别读取后台基础费用。
- 非会员、过期会员、规则停用、规则未生效均按原价。
- 三种会员等级按各自规则计算。
- 最低实扣和优惠上限正确。
- 发布、两种申请入口、查看联系方式的余额校验与流水值等于实际报价。
- 重复申请与重复解锁不重复扣费。
- 后台聚合保存具备事务性和并发版本校验。

### 15.2 管理后台

- 三个中文场景标签页均能加载和保存。
- 等级、场景、权益类型、状态均不展示英文编码。
- 基础费用和会员规则校验正确。
- 保存失败和版本冲突反馈正确。

### 15.3 小程序

- 发布页移除表单中部费用卡片，底栏展示真实原价、优惠和实扣。
- 报名页展示余额、真实原价、优惠和实扣，主题色正确，按钮宽度为 45%。
- 无优惠、报价加载、报价失败、余额不足状态正确。
- 查看联系方式弹窗展示报价，已解锁时不再扣费。
- 小屏幕与底部安全区下无文本溢出、按钮遮挡和内容遮挡。

## 16. 安全与审计

- 后台接口继续使用现有管理员鉴权。
- 保存前记录操作人、修改前值、修改后值、场景和时间。
- 拍豆流水保留场景与实际扣除，便于对账。
- 日志不记录用户令牌、联系方式或其他敏感数据。

## 17. 回滚策略

1. 前端可回滚到旧页面，后端统一报价接口继续兼容发布约拍。
2. 后端扣费入口回滚时恢复当前 3 拍豆常量，数据库配置表保留但不再读取。
3. 后台回滚到旧权益规则表格时，现有规则接口仍可使用。
4. 新增场景规则和配置表不立即删除，避免回滚期间丢失运营配置。

## 18. 验收标准

1. 后台权益配置按三个中文场景切换，表格无英文业务编码。
2. 每个场景均可保存基础消耗与三种会员等级优惠。
3. 三个扣费入口实际扣费与统一报价一致。
4. 发布页与报名页费用摘要和提交按钮同排，按钮宽度为 45%。
5. 两个页面均展示原价、会员优惠与实扣；报名页保留余额。
6. 查看联系方式展示同一报价，并避免重复扣费。
7. 余额不足提示使用实际扣除拍豆。
8. 后台修改配置后，新报价无需重新发布客户端即可生效。
