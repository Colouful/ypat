# 新版管理后台内测数据工厂设计

日期：2026-07-06

分支：`codex/admin-internal-seeding`

## 背景

新版管理后台需要一个独立维护的内测数据能力，用于单个或批量创建内测用户，并为这些用户创建约拍信息、发布约拍、约摄影师快捷场景和作品内容。已有新版管理后台中存在后台代发约拍能力，但它和真实用户数据缺少清晰标识，不适合长期作为内测数据生产工具。

本次设计采用独立的 `内测数据工厂` 模块，面向新版管理后台开发，不修改旧版后台 `backend/system-web`。

## 范围

本次包含：

1. 在新版管理后台新增 `内测数据` 菜单组。
2. 新增 `内测资源管理` 页面，用一张资源表维护图片和视频资源。
3. 新增 `内测数据生成` 页面，支持单个或批量创建内测用户、约拍和作品。
4. 支持发布状态选择：默认待审核，也可直接生成审核通过内容。
5. 给用户、约拍和作品主表增加内测数据标识和批次号。
6. 支持按批次或筛选条件软清理内测数据。

本次不包含：

1. 不修改旧版后台页面和控制器。
2. 不做 CSV 导入。
3. 不做资源文件上传。资源 URL 由运营自行上传后维护到资源管理。
4. 不物理删除内测数据。
5. 不把约拍和作品合并成新的内容模型。

## 方案取舍

采用独立模块方案，而不是增强现有后台代发约拍页。

优点：

- 用户、资源、约拍、作品的内测能力边界清晰。
- 可以独立筛选、统计、追加生成和软清理。
- 不污染现有审核、查询和代发页面。
- 更适合后续扩展视频资源、批次管理和更多模板。

代价：

- 需要新增资源表、主表标识字段、几组后台接口和两个新版管理后台页面。

## 菜单设计

新增菜单组：`内测数据`。

子页面：

- `内测资源管理`：维护图片和视频资源，供生成内测用户头像、约拍图片、作品图片和作品视频使用。
- `内测数据生成`：创建内测用户池，生成约拍和作品，查看批次结果并执行软清理。

现有 `后台代发约拍` 页面保留不动。

## 数据库设计

### 内测资源表

新增表：`t_internal_test_resource`。

字段：

| 字段 | 含义 |
| --- | --- |
| `id` | 主键 |
| `media_type` | 媒体类型：`image` / `video` |
| `usage_type` | 用途类型：`avatar` / `ypat` / `work` |
| `style_code` | 风格编码，对齐约拍和作品风格 |
| `url` | 资源 URL |
| `title` | 标题 |
| `description` | 描述 |
| `profession` | 职业，可空 |
| `city` | 城市，可空 |
| `status` | 状态：`enabled` / `disabled` |
| `sort_no` | 排序号 |
| `remark` | 备注 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

资源表用 `media_type` 区分图片和视频，管理后台通过 tab 控制展示。

### 主表标识字段

给三张主表增加字段：

- `t_user.data_flag`
- `t_ypat_info.data_flag`
- `t_work.data_flag`

取值：

- `real`：真实数据，默认值。
- `internal_test`：内测数据。

给三张主表增加批次字段：

- `t_user.internal_batch_no`
- `t_ypat_info.internal_batch_no`
- `t_work.internal_batch_no`

`internal_batch_no` 用来追踪一次生成的数据，便于筛选、回溯和软清理。真实数据批次号为空。

## 后端设计

新增后台接口前缀：`/admin/internal-test/*`。

涉及模块：

- `backend/system-object`：新增资源、生成配置、批次结果等传输对象。
- `backend/system-domain`：新增内测资源和内测数据生成服务。
- `backend/system-restapi`：暴露内部服务接口。
- `backend/system-wap`：暴露新版管理后台接口。

不修改 `backend/system-web`。

### 资源管理接口

- `GET /admin/internal-test/resources`
  - 按媒体类型、用途、风格、职业、城市、状态和关键词分页查询资源。
- `POST /admin/internal-test/resources`
  - 新增资源。
- `POST /admin/internal-test/resources/update`
  - 编辑资源。
- `POST /admin/internal-test/resources/status`
  - 启用或停用资源。

### 数据生成接口

- `GET /admin/internal-test/users`
  - 只查询内测用户池。
- `POST /admin/internal-test/users/create`
  - 单个或批量创建内测用户。
- `POST /admin/internal-test/generate`
  - 给新用户或已有内测用户生成约拍和作品。
- `GET /admin/internal-test/batches`
  - 查看生成批次摘要。
- `POST /admin/internal-test/cleanup`
  - 按批次、用户或筛选条件软清理内测数据。

### 生成规则

- 手机号使用内测专用生成规则，避免与真实手机号冲突。
- 用户昵称、职业、性别和城市由表单模板生成。
- 头像从资源表按 `usage_type=avatar` 匹配。
- 约拍资源从资源表按 `usage_type=ypat`、风格、媒体类型匹配。
- 作品资源从资源表按 `usage_type=work`、风格、媒体类型匹配。
- 资源不足时返回明确错误，不生成缺图或缺视频数据。
- 生成状态默认待审核，可选择直接审核通过。
- 所有生成数据写入 `data_flag=internal_test` 和同一个 `internal_batch_no`。
- 约摄影师作为约拍快捷模板，底层仍写入 `t_ypat_info`。

### 软清理规则

软清理只允许处理 `data_flag=internal_test` 的数据。

- 内测用户：改为禁用或不可用状态。
- 内测约拍：改为下架或审核不通过状态。
- 内测作品：改为已下架状态。
- 内测资源：改为停用状态。

真实数据不参与软清理。

## 新版管理后台设计

### 内测资源管理

页面结构：

- 顶部 tab：图片 / 视频。
- 筛选项：用途、风格、职业、城市、状态、关键词。
- 表格字段：预览、标题、媒体类型、用途、风格、职业、城市、状态、排序、创建时间。
- 操作：新增、编辑、启用、停用。

新增和编辑弹窗维护 URL 与元信息，不处理上传。

### 内测数据生成

页面分为三块：

1. 生成配置
   - 生成模式：新建用户并生成 / 给已有内测用户追加。
   - 数量：第一版单次最多 50 个用户。
   - 用户模板：城市、职业、性别、昵称前缀。
   - 内容类型：约拍、作品、两者都生成。
   - 快捷模板：发布约拍、约摄影师、约模特等。
   - 发布状态：默认待审核，可选已通过。

2. 资源选择
   - 根据风格、用途和媒体类型自动匹配资源。
   - 支持手动指定头像资源、约拍图片资源、作品图片或视频资源。
   - 提交前校验资源是否足够。

3. 结果与批次
   - 展示批次号、创建用户数、约拍数、作品数和失败原因。
   - 支持跳转查看内测用户池。
   - 支持按批次执行软清理。

交互约束：

- 提交生成前二次确认。
- 确认文案必须提示“仅生成内测数据”。
- 清理前二次确认。
- 清理确认文案必须提示“仅影响内测数据”。

## 数据流

1. 运营自行上传素材，获得资源 URL。
2. 在 `内测资源管理` 中维护资源 URL、媒体类型、用途、风格和状态。
3. 在 `内测数据生成` 中选择生成模式、数量、风格、状态和资源。
4. 后端校验资源和模板。
5. 后端创建内测用户、约拍和作品，并写入批次号。
6. 前端展示批次结果。
7. 如需回收，按批次或条件执行软清理。

## 错误处理

- 资源不足：返回缺少的用途、风格和媒体类型。
- 手机号冲突：跳过冲突号码并继续生成可用号码；达到重试上限后失败。
- 生成数量超过上限：前端和后端都拦截。
- 已停用资源：不能被生成任务选用。
- 清理条件命中真实数据：后端忽略真实数据，并返回真实数据未处理数量。
- 部分生成失败：返回批次号、成功数量和失败明细。

- 新版管理后台出现 `内测数据` 菜单组。
- 可以维护图片和视频内测资源，一张表通过媒体类型区分。
- 可以单个或批量创建内测用户。
- 可以生成约拍、约摄影师快捷模板和作品内测内容。
- 可以给已有内测用户追加约拍和作品。
- 可以选择默认待审核或直接审核通过。
- 所有内测用户、约拍和作品都有 `data_flag=internal_test` 和批次号。

- 软清理不影响真实数据。
- 旧版后台 `backend/system-web` 没有改动。

## 风险与控制

- 风险：内测数据误入真实数据清理范围。
  - 控制：所有清理条件必须附加 `data_flag=internal_test`。
- 风险：资源不足导致生成内容质量差。
  - 控制：提交前和后端执行前都校验资源数量。
- 风险：批量生成过多影响数据库和审核列表。
  - 控制：第一版单次最多 50 个用户，并保留后端上限。
- 风险：直接审核通过生成内容污染用户端。
  - 控制：生成表单默认待审核，选择已通过必须二次确认。
- 风险：旧版后台被误改。
  - 控制：实现计划明确排除 `backend/system-web`。
