# 约拍申请与消息展示链路修复设计

## 背景与根因

用户提交普通约拍后，申请记录已经由后端保存为 `MessType.send`（约拍申请消息，值为 `1`），但新版以下两个页面均查询了 `/my/ypat/send/list`（申请反馈消息接口）：

- `pages-sub/ypat/my-apply`（我的申请页面）
- `pages/message/index`（消息页面）的“申请的约拍”标签

该接口沿用旧版语义，只查询接收人为当前用户且类型为 `MessType.view`（已查看反馈消息，值为 `4`）的记录。此类记录只有约拍发布者查看申请人的联系方式后才生成，所以申请刚提交时接口返回空列表。

旧版小程序使用 `/my/ypat/app/list`（我的申请列表接口）查询当前用户提交的申请。该接口按 `sendperid=当前用户`（发送人为当前用户）和 `type=1`（约拍申请）筛选，返回关联的约拍详情。

本地接口证据表明，用户 4 的 `/my/ypat/app/list`（我的申请列表接口）返回 3 条数据并包含约拍 5，而 `/my/ypat/send/list`（申请反馈消息接口）和 `/my/ypat/rec/list`（收到申请接口）均为空。因此故障发生在新版前端的数据源选择，不是提交或落库失败。

## 目标

1. “我的申请”页面展示当前用户已经提交的全部普通约拍申请。
2. 消息页面“申请的约拍”标签展示相同的申请集合，提交后立即可见。
3. 消息页面“收到的约拍”继续展示别人发给当前用户的约拍申请。
4. 保留后端“发布者查看联系方式后生成反馈消息”的既有逻辑，不改变扣豆、消息创建或未读计算。
5. 兼容旧接口的 `pages/totals`（页数/总数）分页字段，避免列表只能显示第一页或错误判定没有更多数据。

## 方案

### 接口边界

在 `frontend/src/api/modules/ypat.ts`（约拍接口模块）增加 `getMyApplicationList`（获取我的申请列表）：

- 使用 GET（查询请求）调用 `/my/ypat/app/list`（我的申请列表接口）。
- 返回类型为 `PageResult<YpatInfo>`（约拍详情分页结果）。
- 在接口模块内将旧分页响应转换为前端统一分页结构。

完善 `normalizePageResult`（分页结果标准化方法），当响应包含 `content`（内容列表）时同时兼容：

- `totalElements`（标准总条数）或 `totals`（旧版总条数）。
- `totalPages`（标准总页数）或 `pages`（旧版总页数）。
- 缺少当前页编号时，使用请求中的页码作为调用方回退值，不在适配器中推测业务页码。

### 我的申请页面

`pages-sub/ypat/my-apply.vue`（我的申请页面）使用 `getMyApplicationList`（获取我的申请列表），列表项恢复为 `YpatInfo`（约拍详情）模型：

- 封面使用 `pics[0]`（第一张约拍图片）。
- 标题使用 `targetTxt`（约拍对象说明）。
- 内容使用 `describ`（约拍描述）。
- 发布者信息来自 `userQo`（发布者信息）。
- 地点使用城市和区域。
- 时间使用 `timeStr`（相对时间）或发布时间。
- 点击后使用约拍编号进入 `/pages-sub/ypat/detail`（约拍详情页面）。

### 消息页面

消息页面保留两个标签，但明确数据职责：

- “收到的约拍”：继续调用 `/my/ypat/rec/list`（收到申请接口），数据类型为 `MessInfo`（消息详情）。
- “申请的约拍”：改用 `/my/ypat/app/list`（我的申请列表接口），数据类型为 `YpatInfo`（约拍详情）。

页面根据当前标签分别渲染消息卡片和申请卡片，不将两种不兼容的数据强制伪装成同一类型。申请卡片展示发布者头像、昵称、约拍描述、地点和相对时间，点击约拍编号直接进入约拍详情。

原“申请未读”只统计 `MessType.view`（已查看反馈消息），与“已提交申请数量”语义不符。改为“申请总数”，数值取标准化后的申请列表 `totalElements`（总条数）；申请标签上的数字同样表示申请总数，不再表示未读反馈。

### 错误与刷新

- 页面进入、下拉刷新和切换标签时重新请求第一页。
- 加载失败时保留现有提示，不用空数组掩盖接口错误。
- 触底分页使用标准化后的总页数判断是否还有下一页。
- 提交页成功返回后，两个目标页面通过现有 `onShow`（页面显示钩子）刷新，无需新增全局状态或事件总线。

## 测试范围

1. 接口契约测试确认 `getMyApplicationList`（获取我的申请列表）使用 GET（查询请求）访问 `/my/ypat/app/list`（我的申请列表接口）。
2. 分页适配测试覆盖 `{ content, pages, totals }`（旧分页结构）到统一分页结构的转换。
3. “我的申请”页面测试确认使用申请接口、约拍详情模型和约拍编号导航。
4. 消息页面测试确认收到标签仍使用收到申请接口，申请标签使用我的申请接口，并且申请总数不再读取旧反馈未读接口。
5. 运行与约拍详情、申请提交、消息导航相关的现有测试，确认没有回归。

## 非目标

- 不修改普通约拍提交和扣豆事务。
- 不修改作品快捷约拍链路。
- 不删除 `/my/ypat/send/list`（申请反馈消息接口）或后端 `MessType.view`（已查看反馈消息）逻辑。
- 不新增数据库表、字段或数据迁移。
- 不调整消息页面整体视觉设计。

## 验收标准

1. 用户提交约拍成功后，进入“我的申请”能看到该约拍。
2. 用户进入消息页面并切换到“申请的约拍”后能看到该约拍。
3. 两个入口点击同一条申请均进入正确的约拍详情。
4. “收到的约拍”仍展示其他用户向当前用户提交的申请。
5. 申请超过一页时能够正常加载下一页。
6. 不影响发布者查看联系方式、扣豆和反馈消息生成逻辑。
