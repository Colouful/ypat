# 新版小程序消息链路升级设计

## 背景

当前消息子系统以 `t_mess_info` 为站内消息真源，微信订阅消息只承担站外通知。新版小程序已经有消息 tab(标签页)、未读数、收到/申请列表、消息详情和查看联系方式流程，但还存在三类问题：

- 订阅授权缺失：新版前端没有完整迁移旧版的 `wx.requestSubscribeMessage`(微信订阅授权) 行为，用户可能从未授权模板，导致微信侧通知无法到达。
- 推送可观测性不足：后端调用微信订阅消息后只写日志，不落库，后台无法判断某次业务事件是否创建了站内消息、是否发起了微信推送、微信返回了什么错误。
- 消息 tab(标签页)体验偏基础：底部消息红点、消息页摘要、未读/已读/联系方式状态和主题色分层还不够清晰。

本轮采用“事件日志 + 轻量体验升级”方案：保留现有 HTTP(超文本传输协议) 拉取架构，不引入 `WebSocket`(网页套接字)，通过新增消息推送日志、后台查询页、小程序订阅授权和消息页视觉优化补齐完整链路。

## 目标

- 新增消息/推送事件记录表，记录站内消息创建事件和微信订阅消息发送事件。
- 后台管理新增“推送记录”页，支持按类型、结果、用户、openid(微信用户标识)、消息 ID、约拍 ID、时间范围查询。
- 现有后台“消息列表”保留并增强，可关联查看对应推送记录。
- 新版小程序补齐关键动作前置订阅授权：发布约拍、申请约拍前请求相关模板授权，H5(网页端) 直接放行。
- 新版小程序消息页新增“开启消息提醒”入口，用户可主动请求消息相关模板授权。
- 新版小程序补齐轻量刷新触发：进入消息页、切换 tab(标签页)、申请成功、查看联系方式成功、底部消息红点均刷新未读数。
- 新版小程序消息页使用主题色做功能分层：收到消息偏绿色，申请/已查看偏蓝色或橙色，未读状态更醒目。

## 非目标

- 不引入 `WebSocket`(网页套接字)、SSE(服务端事件) 或 MQ(消息队列)。
- 不做模板 ID 动态配置和后台模板开关管理。
- 不重构现有 `t_mess_info` 数据结构，不改变未读数主查询逻辑。
- 不修改旧版前端项目 `91pai-master`，旧版仅作为行为参考。
- 不改造短信旁路推送。
- 不下载依赖，不启动完整构建；实现后由用户手动体验验证。

## 总体架构

链路分三层：

1. 站内消息层  
   继续以 `t_mess_info` 为业务真源。约拍申请、查看联系方式写入消息、未读数、消息详情和列表仍走现有服务。新增日志记录作为旁路能力，不改变原业务成功失败判断。

2. 推送记录层  
   新增 `t_message_push_log` 表，记录两类事件：`IN_APP_CREATED`(站内消息已创建) 和 `WECHAT_SUBSCRIBE_SENT`(微信订阅消息已发送或失败)。记录包含业务类型、消息 ID、约拍 ID、发送/接收用户、openid(微信用户标识)、模板 ID、跳转页面、微信响应、成功状态和错误信息。

3. 展示体验层  
   小程序侧负责授权、刷新和视觉优化；后台侧负责审计查询和排错。后台不直接重放推送，避免本轮扩大到补偿队列或人工重试系统。

## 后端数据模型

新增 entity(实体) `MessagePushLog`，对应表 `t_message_push_log`。

字段建议：

| 字段 | 说明 |
| --- | --- |
| `id` | 主键 |
| `event_type` | 事件类型：`IN_APP_CREATED` 或 `WECHAT_SUBSCRIBE_SENT` |
| `business_type` | 业务类型：`send`、`oauth`、`audit`、`order`、`view` |
| `message_id` | 关联 `t_mess_info.id`，可为空 |
| `ypatid` | 关联约拍 ID，可为空 |
| `sendperid` | 发送人用户 ID，可为空 |
| `recperid` | 接收人用户 ID，可为空 |
| `touser_openid` | 微信接收人 openid(微信用户标识)，可为空 |
| `template_id` | 微信订阅模板 ID，可为空 |
| `page` | 订阅消息点击跳转页面 |
| `success` | 是否成功，建议用 `yes`/`no` 对齐既有枚举风格 |
| `wechat_errcode` | 微信返回错误码，可为空 |
| `wechat_errmsg` | 微信返回错误文案，可为空 |
| `response_body` | 微信原始响应，截断存储 |
| `remark` | 站内创建说明、异常说明或业务备注 |
| `created_at` | 创建时间 |

索引：

- `idx_message_push_log_created_at(created_at)`
- `idx_message_push_log_message_id(message_id)`
- `idx_message_push_log_ypatid(ypatid)`
- `idx_message_push_log_openid(touser_openid)`
- `idx_message_push_log_result(event_type, business_type, success)`

SQL(数据库脚本) 放在：

- `backend/dev/mysql/20260709_create_message_push_log.sql`

项目当前未引入 `Flyway`(数据库迁移工具) 或 `Liquibase`(数据库迁移工具)，本轮延续现有手工 SQL(数据库脚本) 做法。

## 后端事件记录

新增 `MessagePushLogService`(消息推送日志服务)，负责提供窄接口：

- `recordInAppCreated(...)`：记录站内消息创建事件。
- `recordWechatSubscribeSent(...)`：记录微信订阅发送结果。
- `findPage(...)`：后台分页查询。
- `stats(...)`：后台统计今日总数、成功数、失败数、失败率。

推荐落点：

- `MessInfoService.myRecAdd`：创建约拍申请站内消息后，记录 `IN_APP_CREATED`，业务类型 `send`。
- `UserService.recordViewWriteMess` 或当前查看联系方式后写 `MessType.view` 的位置：记录 `IN_APP_CREATED`，业务类型 `view`。
- `MypatInfoController.myRecAdd`：调用微信订阅消息后，记录 `WECHAT_SUBSCRIBE_SENT`，业务类型 `send`。
- `YpatInfoController.submit`：发布约拍推管理员后，记录 `WECHAT_SUBSCRIBE_SENT`，业务类型 `order`。
- `OauthController.add`：实名认证提交推管理员后，记录 `WECHAT_SUBSCRIBE_SENT`，业务类型 `order`。
- `AdminYpatController` 和已有审核入口：审核结果推发布者后，记录 `WECHAT_SUBSCRIBE_SENT`，业务类型 `audit`。
- 实名认证审核结果入口：推申请人后，记录 `WECHAT_SUBSCRIBE_SENT`，业务类型 `oauth`。

`WxMessClient.sendMsg`(微信消息发送客户端) 继续返回微信响应体。外层解析 JSON(数据交换格式)：

- `errcode == 0`：记录成功。
- `errcode != 0`：记录失败、错误码和错误信息。
- 抛异常：记录失败，`remark` 写异常摘要。

失败记录不能中断原业务流程，保持现有“微信推送失败不影响主业务”的隔离策略。

## 后台管理接口

新增接口：

### `GET /admin/message-push-log/list`

分页查询推送记录。

查询条件：

- `eventType`
- `businessType`
- `success`
- `messageId`
- `ypatid`
- `sendperid`
- `recperid`
- `openid`
- `dateStart`
- `dateEnd`
- `page`
- `size`

返回 `PageResult`(分页结果)，字段映射为后台展示模型。

### `GET /admin/message-push-log/stats`

返回今日摘要：

- `total`
- `successCount`
- `failedCount`
- `failedRate`
- `wechatTotal`
- `inAppTotal`

### 现有 `GET /admin/mess/list`

保持现有查询能力，并增强展示字段：

- 消息类型。
- 已读状态。
- 联系方式查看状态。
- 创建时间。
- 发送人/接收人 ID。
- 关联约拍 ID。
- 操作列增加“查看推送记录”，按 `messageId` 跳转到推送记录页。

## 后台页面设计

`frontend-admin` 新增独立菜单“推送记录”，建议放在当前“查询系统”下，与“消息列表”相邻。

页面结构：

- 顶部统计卡片：今日事件数、微信发送成功、微信发送失败、失败率。
- 查询区：事件类型、业务类型、结果、消息 ID、约拍 ID、发送人 ID、接收人 ID、openid(微信用户标识)、时间范围。
- 表格区：事件类型、业务类型、消息 ID、约拍 ID、发送/接收用户、openid、模板 ID、跳转页面、结果、微信错误、创建时间。
- 操作：查看原始响应弹窗，复制 openid(微信用户标识)，跳转关联消息列表。

现有“消息列表”页增强：

- 增加类型和状态列。
- 增加“推送记录”操作按钮。
- 查询条件仍保留至少一个条件的限制，避免无条件扫表。

## 小程序订阅授权设计

新增 `frontend/src/utils/subscribe-message.ts`，封装订阅授权逻辑。

职责：

- 调用 `contentApi.getTemplateIds()` 获取后端模板列表。
- 按旧版 ID 规则筛选模板：发布约拍、申请约拍使用模板 0/1/2；管理员消息授权使用模板 3。
- 在 `MP-WEIXIN`(微信小程序平台) 调用 `wx.requestSubscribeMessage`(微信订阅授权)。
- 在 H5(网页端) 或非微信小程序平台直接返回成功。
- 用户拒绝时返回明确结果，不抛出影响业务的未知异常。

接入点：

- 发布约拍确认提交前：请求模板 0/1/2。
- 申请约拍提交前：请求模板 0/1/2。
- 消息页“开启消息提醒”：请求模板 0/1/2/3，或至少请求 0/1/2。

授权策略：

- 关键业务动作前，如果用户拒绝授权，提示“未开启通知也可继续提交”并允许继续提交。这样避免订阅授权成为业务阻断点。
- 消息页主动入口，如果用户拒绝授权，仅提示“可在下次操作时再次开启”，不改变页面状态。

## 小程序消息页视觉与交互

消息页采用功能分层主题：

- 收到的约拍：绿色主色，强调“待处理/收到”。
- 申请的约拍：蓝色或橙色辅助色，强调“我发起/已查看”。
- 未读：红点或绿色实心点 + 未读标签。
- 已读：弱化为灰色文字。
- 已查看联系方式：橙色状态标签。
- 未查看联系方式：绿色状态标签，引导进入详情。

页面内容增强：

- 顶部摘要区展示总未读、收到未读、申请动态数量。
- 消息页内提供“开启消息提醒”按钮或轻量入口。
- tab(标签页) 文案保留“收到的约拍 / 申请的约拍”，旁边显示对应未读角标。
- 卡片展示头像、昵称、城市、时间、消息内容、已读状态、联系方式状态、右侧进入箭头。
- 空态继续提供去首页/广场入口。

底部 `KeepTabBar`(自定义底部导航)：

- 接入 `userStore.unreadCount`，消息 tab(标签页) 显示红点或数量。
- 点击消息 tab(标签页) 时进入消息页后触发未读刷新。

## 轻量刷新策略

不引入 `WebSocket`(网页套接字)，保留 HTTP(超文本传输协议) 拉取，补齐触发点：

- `App.onShow`(应用显示)：继续刷新未读数。
- `pages/message/index.vue onShow`(页面显示)：刷新未读数并刷新当前列表。
- 切换消息 tab(标签页)：刷新对应列表和未读数。
- 申请约拍成功：调用 `userStore.refreshUnreadCount()` 或返回前由消息页刷新。
- 查看联系方式成功：刷新用户信息、刷新未读数、更新当前消息状态。
- 退出登录：清空未读数和消息状态，避免多账号串数据。

`useMessageStore`(消息状态仓库) 和 `userStore.unreadCount` 存在职责重叠。本轮推荐收敛为：

- 全局底部导航红点继续使用 `userStore.unreadCount`。
- 消息页内部使用本地 `recUnreadCount`、`sendUnreadCount` 或轻量组合函数获取分项数量。
- 如保留 `useMessageStore`，只用于消息页分项计数，不再和全局未读并行维护同一份状态。

## 兼容性与安全

- 后端新增日志失败不能影响原业务事务，必要时捕获日志写入异常。
- `response_body` 需要截断，避免微信响应或异常堆栈过长。
- 后台 openid(微信用户标识) 展示建议部分脱敏，但支持复制完整值用于排查。
- 订阅授权只在用户主动点击业务按钮或提醒入口时触发，符合微信小程序要求。
- 后台查询默认按时间倒序，分页必须限制 `size` 上限。
- 旧版 `91pai-master` 不修改。

## 验证策略

按用户要求，不下载依赖、不启动完整构建，由用户手动体验验证。实现阶段做改动相关的轻量验证：

- 后端：补 `MessagePushLogService` 单元测试或源码契约测试，覆盖站内创建记录、微信成功记录、微信失败记录、统计查询。
- 后端：检查 `WxMessClient.sendMsg` 返回值解析不会影响现有推送调用。
- SQL(数据库脚本)：检查表名、字段名、索引名与 entity(实体) 一致。
- 小程序前端：补订阅授权工具测试，覆盖微信环境成功、拒绝、非微信平台放行。
- 小程序前端：补消息导航/未读刷新相关测试，确保收到消息仍进消息详情，申请消息仍进约拍详情。
- 后台管理：补 API(接口)模块测试，确认 `/admin/message-push-log/list`、`/admin/message-push-log/stats` 路径和查询参数正确。
- 后台管理：补菜单和页面源码级检查，确认“消息列表”和“推送记录”入口存在。

## 风险和处理

- **微信推送失败不应阻断主业务**：日志记录也使用捕获保护，保持现有业务隔离。
- **日志字段拿不到 messageId**：如果旧方法没有返回站内消息 ID，先记录可获得的 `ypatid`、发送/接收用户和业务类型；后续若需要强关联，再单独调整 domain(领域服务) 返回值。
- **订阅授权被用户拒绝**：不阻断发布或申请，只提示用户下次可开启通知。
- **后台无条件查询压力**：推送记录页允许默认查最近一天，消息列表继续要求至少一个业务条件。
- **状态双写混乱**：全局红点使用 `userStore.unreadCount`，消息页分项计数局部维护，减少双 store(状态仓库) 不一致。
- **模板 ID 仍是硬编码来源**：本轮保留 `/tmplid/list`，不做模板配置化。
