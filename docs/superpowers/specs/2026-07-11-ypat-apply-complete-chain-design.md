# 普通约拍申请完整链路修复设计

## 背景

用户在 `pages-sub/work/apply`（约拍申请页）从普通约拍详情进入后，填写信息并点击提交，页面提示“约拍信息不完整”。网络面板只看到 `/user/get?id=4`（用户查询接口）返回成功，多次点击也重复出现该成功请求。

实际请求和数据库证据如下：

- `/user/get`（用户查询接口）只是申请页提交前的拍豆余额刷新，不是约拍提交接口。
- `/ypat/get?id=5`（约拍详情接口）返回 `userQo.id=17`，但顶层 `userid=null`。
- 申请页只读取顶层 `userid`，因此在真正调用 `/my/ypat/rec/add`（新增约拍申请接口）前返回“约拍信息不完整”。
- 数据库中用户 `4` 已分别对约拍 `4`、`5` 提交过一次，现有余额为 `5`；本次重复点击没有新增消息或扣豆流水。
- 普通约拍详情接口只返回收藏状态，没有返回当前用户是否已经申请。
- 后端已有重复申请查询，但“查询后写入”之间没有并发锁，两个同时到达的请求仍可能同时通过检查。

## 目标

- 修复普通约拍申请页因发布者编号缺失而无法提交的问题。
- 已申请过同一约拍的用户不能再次提交，详情页和申请页都要有明确状态。
- 快速连续点击、旧客户端或多设备并发请求不能产生重复扣豆和重复约拍消息。
- 发送者身份由 Token（登录令牌）确定，接收者身份由后端约拍记录确定，不信任客户端传入的用户编号。
- 保持作品快捷约拍与普通约拍的防重复策略一致。

## 非目标

- 不修改拍豆消耗数量和会员优惠规则。
- 不新增数据库字段或迁移脚本。
- 不修改旧版 `91pai-master`（旧版小程序）业务。
- 不重构通用请求层或消息中心。
- 不改变约拍发布、审核、收藏和阅读量逻辑。

## 方案对比

### 方案 A：前端发布者编号兜底

申请页从 `userid` 或 `userQo.id` 中取发布者编号。

优点是改动最小，可以消除“约拍信息不完整”。缺点是仍然把接收者身份交给客户端，详情页没有已申请状态，也不能解决并发重复提交。

### 方案 B：完整链路修复（采用）

后端详情明确返回发布者编号和已申请标记；客户端申请参数只保留约拍编号与内容；后端根据登录用户和约拍记录确定双方身份；普通约拍和作品快捷约拍在重复检查前锁定申请用户；前端详情页和申请页都消费已申请状态。

该方案无需数据库迁移，覆盖当前错误、重复入口和并发一致性，改动范围仍限定在约拍链路。

### 方案 C：数据库唯一索引

在消息表增加发送者、约拍、消息类型的唯一索引。

优点是数据库层保护最强。缺点是需要检查存量重复数据、编写迁移和回滚脚本，并评估其他消息类型是否允许重复。本次使用现有悲观锁即可实现相同业务目标，不扩大数据库变更范围。

## 接口契约

### 约拍详情

`/ypat/get`（约拍详情接口）继续返回现有结构，并保证：

- `userid`：约拍发布者编号，来源于 `YpatInfo.user.id`（约拍实体关联用户编号）。
- `msgflag`：当前登录用户是否已提交该约拍，`1` 表示已提交，`0` 表示未提交。
- 未登录用户的 `msgflag` 返回 `0`。

`YpatInfoQo`（约拍详情传输对象）已有 `msgflag` 字段但缺少访问方法，本次补齐 getter/setter（读取与写入方法），并在详情服务中设置。

### 新增约拍申请

前端 `YpatApplyParams`（约拍申请参数）只保留：

```ts
interface YpatApplyParams {
  ypatid: number
  content?: string
}
```

服务端处理规则：

- `sendperid`（发送者编号）由 WAP（小程序服务）根据 Token（登录令牌）覆盖设置。
- `recperid`（接收者编号）由领域服务从 `YpatInfo.user`（约拍关联用户）获取。
- 客户端不再发送或决定双方用户编号。

## 后端设计

### 详情状态

在 `YpatInfoService.findById`（约拍详情领域方法）中：

1. 复制约拍实体后，显式把关联用户编号写入 `userid`。
2. 当前用户存在时，复用 `MessInfoRepository.countSend`（统计已发送约拍消息）查询。
3. 数量大于 0 时设置 `msgflag=1`，否则设置 `msgflag=0`。
4. 收藏状态逻辑保持不变。

### 普通约拍并发保护

在 `MessInfoService.myRecAdd`（新增普通约拍申请领域方法）中：

1. 使用 `UserRepository.findByIdForUpdate`（按编号查询并加写锁）加载申请用户。
2. 在持有用户行悲观写锁期间执行重复消息查询、余额校验、扣豆、流水和消息写入。
3. 同一用户的并发申请串行执行；第二个请求会在第一个事务提交后看到已存在记录并返回 `1006`（数据已存在）。
4. 增加用户、约拍和发布者不存在时的明确业务异常，避免空指针错误。

### 作品快捷约拍并发保护

在 `WorkService.quickApply`（作品快捷约拍领域方法）中，同样使用 `findByIdForUpdate`（按编号查询并加写锁）加载申请用户，保证重复查询和扣豆位于同一锁范围内。现有 `countSendByWorkId`（按作品统计已发送约拍）校验保持不变。

## 前端设计

### 约拍详情页

- `YpatInfo`（约拍详情类型）增加 `msgflag`（已申请标记）。
- `YpatDetailView`（约拍详情视图）根据 `msgflag` 计算已申请状态。
- 未申请时显示“立即约拍”；已申请时显示不可点击的“已约拍”。
- `apply`（申请动作）再次检查已申请状态，避免通过事件直接触发跳转。
- 发布者编号读取兼容 `userid` 和 `userQo.id`，保证旧后端响应下的主页跳转和本人约拍判断正常。

### 返回详情刷新

`pages-sub/ypat/detail.vue`（约拍详情页）在 `onShow`（显示钩子）中调用子组件公开的 `load`（加载方法）。首次进入仍由现有编号监听加载；从申请页返回时重新读取 `msgflag`，更新按钮状态。

### 申请页

- 普通约拍详情加载后读取 `msgflag`。
- 已申请时按钮显示“已约拍”并禁用。
- 提交动作开始后再次加载详情确认最新 `msgflag`，覆盖直接链接、页面停留期间其他设备已提交等情况。
- 申请参数只发送约拍编号和填写内容，不再要求 `userid`、`sendperid` 或 `recperid`。
- 保留现有提交锁、余额刷新、订阅消息和成功返回逻辑。

## 错误处理

- 详情加载失败时不提交，并保留现有错误提示。
- 已申请时提示“你已提交过该约拍”，不调用新增申请接口。
- 旧客户端或并发请求绕过前端时，后端仍返回 `1006`（数据已存在），且不会再次扣豆。
- 余额不足、未实名、未缴保证金继续使用现有业务错误处理。
- 用户、约拍或发布者数据缺失时返回明确业务错误，不产生空指针异常。

## 测试与验收

按 test-driven-development（测试驱动开发）执行：

- 后端源码契约测试覆盖 `userid`、`msgflag` 和悲观锁调用。
- 约拍详情组件测试覆盖未申请跳转、已申请禁用、发布者编号兼容。
- 约拍详情页面测试覆盖返回页面后的重新加载。
- 申请页测试覆盖移除客户端身份参数、已申请拦截和提交前重新检查。
- 只运行相关 Vitest（前端单元测试）、源码断言、ESLint（代码规范检查）和差异格式检查。
- 不下载依赖、不执行完整构建，由用户完成微信开发者工具手动验收。

手动验收标准：

1. 用户未申请普通约拍时，可以填写并成功提交一次。
2. 提交后只新增一条约拍消息、一条扣豆流水，拍豆只扣一次。
3. 返回详情页后按钮显示“已约拍”，不能再次进入申请页。
4. 直接再次打开申请页时显示已约拍状态，不能提交。
5. 快速连续点击或两个客户端并发提交时，最多成功一次。
6. 作品快捷约拍继续保持首次成功、重复提交被阻止。

## 风险控制

- 不修改消息表结构，避免数据库迁移风险。
- 悲观锁仅锁定当前申请用户行，锁范围覆盖短事务，不影响其他用户提交。
- 后端继续保留重复查询，前端状态不作为唯一安全边界。
- 所有修改限定在普通约拍详情、申请和两类约拍并发保护，不处理需求外逻辑。
