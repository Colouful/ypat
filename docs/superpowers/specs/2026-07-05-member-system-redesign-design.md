# 会员系统规则化闭环设计

日期：2026-07-05
工作树：`/Users/lizhenwei/workspace/vueworkspace/ypat-workspace`
分支：`dev6`

## 背景

旧版 `91pai-master(旧版前端)` 没有完整会员模块。新版 `frontend(新版小程序前端)` 已经出现 `pages-sub/user/member/index(会员中心页)`、`memberStore(会员状态仓库)`、会员订单页和“我的页”会员入口；`backend(后端)` 也已有 `/member/plans`、`/member/status`、会员下单和订单状态等雏形。

当前能力尚未闭环：

- `GET /member/plans` 在本地返回 `401 unauthorized(未授权)`，因为后端安全白名单未放开该公开接口，而前端按公开接口不带 `Token(登录令牌)` 请求。
- `frontend-admin(管理后台前端)` 目前没有会员套餐、权益配置、会员用户、会员订单和操作日志页面。
- `docs/sql/pending/V_pending_member.sql(待执行会员数据库脚本)` 仍标注为待执行，套餐价格是测试种子，正式价格需要运营后台配置。
- 当前会员页面较基础，未明确展示“会员提交约拍可减免拍拍豆”的核心价值。

本设计选择“规则化会员闭环”方案：套餐负责售卖价格和时长，等级决定权益，权益按业务场景配置。一期只上线 `BASIC(基础会员)`，但按等级建模，为后续 `PRO(高级会员)` 或更多业务场景留扩展空间。

## 目标

1. 建立完整会员闭环：套餐配置、权益规则、微信支付开通、后台人工调整、会员状态、订单和日志。
2. 支持后台配置“提交约拍”会员优惠。例如原价扣 5 个拍拍豆，后台配置会员优惠 2 个，会员实扣 3 个。
3. 支持后台动态配置套餐：名称、时长、售价、划线价、赠送拍拍豆、绑定等级、上下架和排序。
4. 支付成功后自动开通或续期会员，自动发放套餐赠送拍拍豆，并写入 `record(收支记录)`。
5. 后台允许赠送、延长、取消会员，必须填写原因并写操作日志。
6. 后台不允许手动修改支付订单状态、微信流水和用户拍拍豆余额。

## 不做内容

- 一期不做自动续费。
- 一期不做自动退款。
- 一期不售卖多个会员等级，只启用 `BASIC(基础会员)`。
- 一期不接入“报名约拍”“查看联系方式”等场景的真实扣减，只预留规则配置能力。
- 后台不提供删除会员操作日志能力。

## 领域模型

### 会员套餐

`member_plan(会员套餐)` 由管理后台维护，面向用户端展示。核心字段：

| 字段 | 说明 |
| --- | --- |
| `code` | 套餐编码，例如 `MONTH`、`SEASON`、`YEAR` |
| `name` | 套餐名称 |
| `durationDays` | 有效天数 |
| `priceFen` | 售价，单位分 |
| `originPriceFen` | 划线价，单位分 |
| `giftPpd` | 开通后赠送拍拍豆数量 |
| `levelCode` | 绑定会员等级，一期固定为 `BASIC` |
| `recommended` | 是否推荐展示 |
| `status` | 上架或下架 |
| `sortNo` | 展示排序 |

套餐编辑只影响新订单。历史订单必须保存下单时的套餐快照，避免后续改价影响对账。

### 会员等级

`member_level(会员等级)` 一期内置 `BASIC(基础会员)`：

| 字段 | 说明 |
| --- | --- |
| `code` | 等级编码，一期为 `BASIC` |
| `name` | 展示名，例如基础会员 |
| `status` | 是否启用 |
| `description` | 说明文案 |

即使一期只有一个等级，也保留等级边界。这样未来增加 `PRO(高级会员)` 时，只需要新增等级、套餐和权益规则，不必重做订单和用户会员状态。

### 权益规则

`member_benefit_rule(会员权益规则)` 按“等级 + 业务场景”配置：

| 字段 | 说明 |
| --- | --- |
| `levelCode` | 会员等级，一期为 `BASIC` |
| `scene` | 业务场景，一期真实接入 `SUBMIT_YPAT(提交约拍)` |
| `benefitType` | 权益类型，一期为 `PPD_DISCOUNT(拍拍豆减免)` |
| `discountPpd` | 优惠拍拍豆数量 |
| `minActualPpd` | 最低实扣，默认 0 |
| `effective` | 是否已接入业务生效 |
| `status` | 启用或停用 |
| `description` | 后台说明 |

计算规则：

```text
actualPpd = max(minActualPpd, originalPpd - discountPpd)
```

示例：系统原价 5，`BASIC(基础会员)` 优惠 2，最低实扣 0，则实扣 3。

### 用户会员状态

`user_member(用户会员状态)` 是判断用户是否会员的事实源：

| 字段 | 说明 |
| --- | --- |
| `userId` | 用户 ID |
| `levelCode` | 当前等级，一期为 `BASIC` |
| `expireAt` | 到期时间 |
| `sourceOrderNo` | 最近一次支付开通或续费订单号 |
| `status` | 有效、取消等状态 |
| `updatedAt` | 更新时间 |

用户是否享受权益由 `expireAt`、`status` 和 `levelCode` 共同判断，不从历史订单反推。

### 会员订单

`member_order(会员订单)` 保存支付订单和套餐快照：

| 字段 | 说明 |
| --- | --- |
| `outTradeNo` | 商户订单号 |
| `userId` | 用户 ID |
| `planId` | 套餐 ID |
| `planCodeSnapshot` | 套餐编码快照 |
| `planNameSnapshot` | 套餐名称快照 |
| `levelCodeSnapshot` | 绑定等级快照 |
| `priceFen` | 下单锁定价 |
| `originPriceFen` | 下单划线价快照 |
| `durationDays` | 下单锁定有效天数 |
| `giftPpd` | 下单锁定赠送拍拍豆数量 |
| `status` | 待支付、已支付、已取消、已退款 |
| `wxTransactionId` | 微信支付流水 |
| `paidAt` | 支付完成时间 |

后台订单页面只读展示这些字段，不提供手动改支付状态。

### 会员操作日志

`member_operation_log(会员操作日志)` 记录支付开通和后台人工操作：

| 字段 | 说明 |
| --- | --- |
| `userId` | 被操作用户 |
| `operatorId` | 操作者，支付回调可为空或系统用户 |
| `actionType` | 支付开通、后台赠送、后台延长、后台取消 |
| `reason` | 人工操作原因，后台操作必填 |
| `beforeValue` | 操作前状态快照 |
| `afterValue` | 操作后状态快照 |
| `sourceOrderNo` | 关联会员订单号，可空 |
| `createdAt` | 操作时间 |

日志只允许新增和查询，不允许删除。

## 后端设计

### 用户端接口

| 方法 | 路径 | 鉴权 | 用途 |
| --- | --- | --- | --- |
| `GET` | `/member/plans` | 否 | 套餐列表，公开展示 |
| `GET` | `/member/status` | 是 | 当前用户会员状态 |
| `GET` | `/member/benefit/quote` | 是 | 指定场景的权益预估 |
| `POST` | `/member/order/create` | 是 | 创建会员订单并发起微信统一下单 |
| `GET` | `/member/order/status` | 是 | 轮询会员订单状态 |
| `GET` | `/member/orders` | 是 | 当前用户会员订单分页 |
| `POST` | `/member/order/cancel` | 是 | 取消待支付会员订单 |

`GET /member/plans` 必须加入 `WebSecurityConfig(安全配置)` 的 `GET permitAll(公开白名单)`，前端继续使用 `withToken=false(不带登录令牌)`。`/member/status` 等用户相关接口继续要求 `Token(登录令牌)`。

`GET /member/benefit/quote` 请求参数：

| 参数 | 说明 |
| --- | --- |
| `scene` | 场景编码，一期为 `SUBMIT_YPAT` |

返回字段：

| 字段 | 说明 |
| --- | --- |
| `scene` | 场景 |
| `memberActive` | 当前会员是否生效 |
| `levelCode` | 当前会员等级 |
| `originalPpd` | 原价拍拍豆 |
| `discountPpd` | 会员优惠拍拍豆 |
| `actualPpd` | 实扣拍拍豆 |
| `ruleEffective` | 规则是否已接入生效 |

预估接口只用于展示，不扣减余额。真实提交约拍时，后端重新调用同一个 `BenefitCalculator(权益计算器)` 计算并扣款。

### 管理后台接口

后台接口统一挂在 `/admin/member/**`，需要管理员 `Token(登录令牌)`。

套餐管理：

- `GET /admin/member/plans`
- `POST /admin/member/plans`
- `PUT /admin/member/plans/{id}`
- `POST /admin/member/plans/{id}/up-down`

权益配置：

- `GET /admin/member/benefit-rules`
- `PUT /admin/member/benefit-rules/{id}`

会员用户：

- `GET /admin/member/users`
- `GET /admin/member/users/{userId}`
- `POST /admin/member/users/{userId}/grant`
- `POST /admin/member/users/{userId}/extend`
- `POST /admin/member/users/{userId}/cancel`

会员订单：

- `GET /admin/member/orders`
- `GET /admin/member/orders/{id}`

操作日志：

- `GET /admin/member/logs`

后台人工操作必须填写 `reason(原因)`。后台人工赠送、延长、取消会员只影响 `user_member(用户会员状态)`，并写 `member_operation_log(会员操作日志)`。不允许后台直接修改 `member_order(会员订单)` 和用户 `ppd(拍拍豆余额)`。

### 支付与账务事务

会员支付流程：

1. 用户选择套餐，调用 `/member/order/create`。
2. 后端校验套餐上架状态，创建待支付订单，并锁定套餐快照。
3. `system-wap(小程序网关)` 调微信统一下单，返回支付参数。
4. 微信支付回调验签成功后，按订单号进入支付成功事务。
5. 事务内只允许 `status=待支付` 的订单变为 `status=已支付`。
6. 开通或续期 `user_member(用户会员状态)`。
7. 如果订单 `giftPpd` 大于 0，增加用户 `ppd(拍拍豆余额)`，并写 `record(收支记录)`。
8. 写 `member_operation_log(会员操作日志)`。

支付回调必须幂等。重复回调返回成功，但不能重复开通会员、不能重复赠送拍拍豆、不能重复写余额流水。

### 提交约拍扣豆

提交约拍使用同一个 `BenefitCalculator(权益计算器)`：

1. 读取提交约拍原价拍拍豆，现有原价为 5。
2. 读取用户会员状态，判断 `BASIC(基础会员)` 是否有效。
3. 读取 `SUBMIT_YPAT(提交约拍)` 的 `PPD_DISCOUNT(拍拍豆减免)` 规则。
4. 计算 `originalPpd(原价)`、`discountPpd(优惠)`、`actualPpd(实扣)`。
5. 检查用户余额是否足够支付 `actualPpd`。
6. 扣减 `actualPpd` 并写收支记录。

前端不得传入优惠金额，后端不得信任前端展示结果。

## 管理后台设计

### 菜单

在 `frontend-admin(管理后台前端)` 新增“会员系统”菜单，包含：

- 套餐管理
- 权益配置
- 会员用户
- 会员订单
- 操作日志

### 套餐管理

列表字段：

- 套餐名称
- 时长
- 售价
- 划线价
- 赠送拍拍豆
- 绑定等级
- 是否推荐
- 上下架状态
- 排序
- 更新时间

操作：

- 新增套餐
- 编辑套餐
- 上架或下架
- 调整排序

编辑套餐只影响新订单。页面需要提示“历史订单保留当时价格和赠送拍拍豆快照”。

### 权益配置

一期展示 `BASIC(基础会员)` 下的场景规则：

- `SUBMIT_YPAT(提交约拍)`：已生效，可编辑优惠拍拍豆、最低实扣、启用状态和说明。
- 其他预留场景：可展示为“未接入”，避免运营误以为配置后立即生效。

权益配置保存后，下一次预估和提交约拍立即使用新规则。

### 会员用户

筛选条件：

- 手机号
- 昵称
- 会员状态
- 到期时间范围

用户详情展示：

- 用户基本信息
- 当前会员等级
- 到期时间
- 最近来源订单
- 会员订单列表
- 操作日志列表

允许操作：

- 赠送 `BASIC(基础会员)` N 天
- 延长当前会员 N 天
- 取消会员资格

每次操作必须填写原因。后台不提供直接修改用户拍拍豆余额入口。

### 会员订单

订单页面只读查询：

- 商户订单号
- 用户信息
- 套餐快照
- 金额
- 赠送拍拍豆
- 支付状态
- 微信流水
- 创建时间
- 支付时间

后台不允许手动改订单状态。如果发生支付异常，只能通过后端补偿脚本或专门运维流程处理，不在运营后台开放。

### 操作日志

日志页面支持按用户、操作类型、操作者和时间范围筛选。日志记录支付开通、后台赠送、后台延长、后台取消等事件。

## 小程序前端设计

### 我的页会员卡片

未开通时展示：

- `爱去拍 · BASIC 会员`
- `提交约拍可省拍拍豆`
- `立即开通`

已开通时展示：

- 当前会员等级
- 有效期
- `去续费`

会员卡片跳转 `/pages-sub/user/member/index(会员中心页)`。

### 会员中心

会员中心结构：

1. 顶部状态区：未开通、已开通、已过期。
2. 套餐区：横向展示后台上架套餐，包含售价、划线价、赠送拍拍豆和推荐标签。
3. 权益区：优先展示“提交约拍减免拍拍豆”和“开通赠送拍拍豆”。
4. 底部固定按钮：开通或续费当前选中套餐。
5. 订单入口：跳转会员订单页。

支付成功后不能直接以 `requestPayment success(微信支付成功回调)` 为开通成功依据。前端必须轮询 `/member/order/status`，后端确认订单已支付后再展示“开通成功”并刷新会员状态。

### 提交约拍页面

发布约拍页面在提交前调用 `/member/benefit/quote?scene=SUBMIT_YPAT` 展示费用明细：

- 原价拍拍豆
- 会员优惠拍拍豆
- 本次实扣拍拍豆

示例文案：

```text
原价：5 拍拍豆
BASIC 会员优惠：-2 拍拍豆
本次实扣：3 拍拍豆
```

非会员展示开通引导：

```text
开通 BASIC 会员，本次提交约拍可少扣 2 拍拍豆。
```

最终提交以后端真实扣减结果为准。如果预估后规则变化，提交接口返回的实扣结果可以与预估不同，前端需要展示后端返回结果或错误提示。

### 前端状态

`memberStore(会员状态仓库)` 负责：

- 缓存会员状态。
- 缓存支付轮询状态。
- 支付成功后刷新会员状态。
- 登录状态变化后刷新或清空会员状态。

套餐列表不需要登录即可展示。会员状态、订单和权益预估需要登录。

## 数据迁移与默认数据

上线前需要从 `docs/sql/pending(V_pending_member.sql)` 迁移为正式 migration(数据库迁移) 或明确执行流程。

默认数据：

- 新增 `BASIC(基础会员)` 等级。
- 新增三档默认套餐，价格和赠送拍拍豆可由运营上线前调整。
- 新增 `SUBMIT_YPAT(提交约拍)` 权益规则，默认优惠数量由运营确认。

历史用户不默认回填会员。后台人工赠送会员可用于活动白名单，不执行全量老用户开通。

5. 验证 `GET /member/plans` 未登录不再返回 401，`/member/status` 未登录仍返回 401。
6. 上小程序会员入口、会员中心和提交约拍费用明细。
7. 通过开关开放“我的页”会员卡片和会员中心入口。

## 回滚与止血

- 小程序入口通过前端开关隐藏。
- 后台下架全部会员套餐，阻止新订单创建。
- 停用 `SUBMIT_YPAT(提交约拍)` 权益规则，恢复原价扣豆。
- 保留会员订单、收支记录和操作日志，不执行删除。
- 如支付回调异常，暂停会员下单入口，保留订单数据供后续补偿。

## 风险与控制

- 风险：套餐列表接口继续 401，导致会员页无法加载。
  - 控制：明确 `GET /member/plans` 是公开接口，加入后端白名单，并加集成测试。
- 风险：支付回调重复导致重复发放会员和拍拍豆。
  - 控制：支付成功事务只允许待支付订单更新为已支付，重复回调不进入发放逻辑。
- 风险：后台人工操作影响账务。
  - 控制：后台只允许赠送、延长、取消会员，不允许手改支付订单和拍拍豆余额。
- 风险：前端优惠展示与真实扣减不一致。
  - 控制：预估和真实提交共用后端 `BenefitCalculator(权益计算器)`；真实提交重新计算。
- 风险：权益规则配置后运营误以为所有场景都生效。
  - 控制：后台规则带 `effective(是否已接入生效)` 标识，一期仅 `SUBMIT_YPAT(提交约拍)` 显示为已生效。
- 风险：历史订单受套餐改价影响。
  - 控制：会员订单保存套餐快照，套餐编辑只影响新订单。
