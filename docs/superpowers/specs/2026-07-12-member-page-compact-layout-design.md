# 会员中心紧凑布局设计

## 目标

优化新版小程序 `pages-sub/user/member/index`(会员中心页面)的信息顺序与首屏密度：把“选择套餐”移动到会员状态卡与会员权益之间，套餐采用双列选择，并完整保留会员权益标题和描述。

页面应自然、美观、易于比较套餐。常见手机首屏优先展示会员状态、三个套餐和会员权益主体；协议与历史订单允许轻微下滑查看，不以牺牲权益信息完整性换取绝对一屏。

## 根因

当前页面纵向过高由源文件布局与信息重复造成，不是编译生成的 `data-v-*`(样式作用域标识)导致：

- 页面顺序为“会员状态卡 → 会员权益 → 套餐”，套餐选择不在决策主流程前部。
- 套餐使用单列大卡片，每张卡同时展示基础信息、权益列表和选择条。
- 三个套餐重复展示权益内容，造成大量纵向占高。
- 会员状态卡上方留白、内部间距和装饰尺寸偏大。

## 布局方案

采用已确认的 A2“平衡双列 · 完整权益”方案：

```text
┌──────────────────────────────┐
│ 返回          会员中心       │
├──────────────────────────────┤
│ 紧凑会员状态卡               │
├──────────────────────────────┤
│ 选择套餐                     │
│ ┌──────────┐ ┌──────────┐   │
│ │ 包月会员 │ │ 包季会员 │   │
│ └──────────┘ └──────────┘   │
│ ┌────────────────────────┐   │
│ │ 包年会员               │   │
│ └────────────────────────┘   │
├──────────────────────────────┤
│ 会员权益                     │
│ ┌──────────┐ ┌──────────┐   │
│ │ 完整描述 │ │ 完整描述 │   │
│ └──────────┘ └──────────┘   │
│ ┌──────────┐ ┌──────────┐   │
│ │ 完整描述 │ │ 完整描述 │   │
│ └──────────┘ └──────────┘   │
├──────────────────────────────┤
│ 当前套餐              开通   │
└──────────────────────────────┘
```

前三个有效套餐按现有排序展示：前两个套餐各占一列；第三个套餐在奇数末项时横跨两列，以避免页面末尾出现不平衡的半行空白。若后台以后配置偶数个套餐，则所有套餐保持双列。

## 视觉调整

### 会员状态卡

- 保留当前深色会员基调、金色会员标识和激活状态差异。
- 减少顶部与标题间距，缩小装饰圆形范围和卡片垂直内边距。
- 保留三个权益关键词，使用更紧凑的标签尺寸。
- 不引入新图片、渐变装饰或额外营销文案。

### 套餐卡

- 卡片圆角不超过现有设计系统卡片圆角。
- 双列卡展示套餐名、有效期、价格、赠送拍拍豆或等级信息。
- 推荐标签固定在右上角；选中状态使用绿色边框、浅色背景和勾选标识。
- 移除每张套餐卡内重复的权益列表与大号“点选此套餐”选择条。
- 第三个横跨套餐采用横向信息布局，价格右对齐。
- 整张套餐卡仍为点击区域，触摸目标不小于 44 像素。

### 会员权益

- 保持 2×2 双列布局。
- 完整保留现有四项权益的图标字、标题和描述，不改业务文案。
- 缩小卡片内边距、图标尺寸与卡片间距，但不截断文字。
- 描述允许自然换行，不设置固定行数或省略号。

## 行为与数据

- 套餐加载、推荐套餐默认选择、手动选择、价格格式化、登录校验、下单与支付逻辑全部保持不变。
- `benefitItems`(权益项目)继续随当前选择套餐更新赠送拍拍豆描述。
- 套餐为空、加载中、会员已生效和未生效状态保持现有逻辑。
- 底部固定操作栏继续显示当前套餐和开通或续费按钮。

## 响应式规则

- 小程序常见宽度下使用两列套餐与两列权益。
- 套餐名、推荐标签和价格不得互相遮挡；长套餐名允许换行。
- 奇数末项横跨两列，避免孤立半宽卡片。
- 安全区域与底部固定操作栏预留保持不变。

## 验证范围

遵循用户要求，不下载依赖、不执行完整构建。验证包括：

- 增加源代码约束测试，确认模板顺序为“状态卡 → 套餐 → 权益”。
- 约束套餐容器使用双列网格，奇数末项横跨两列。
- 约束权益仍保留 2×2 布局及完整描述元素。
- 确认套餐选择、支付和状态加载脚本未被修改。
- 对改动文件执行定向 Vitest(单元测试工具)、ESLint(代码规范检查)和 `git diff --check`(Git 差异格式检查)。
- 启动现有本地开发服务后，使用移动端视口截图检查文字溢出、重叠、首屏高度和选中状态；若服务不可用，则明确记录未完成的视觉验证。
