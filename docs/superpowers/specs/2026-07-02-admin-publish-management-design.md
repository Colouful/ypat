## 背景

新版用户端已经拆分两条发布链路：

- 发布作品：写入 `t_work`，包含作品描述、媒体、标签、作者、状态、计数等字段。
- 发布约拍：写入 `t_ypat_info`，包含约拍对象、风格、收费方式、地区、全国标识、来源作品等字段。

当前管理后台已经有约拍申请列表、约拍查询列表和后台代发页面，但字段、枚举和新版用户端不完全一致；作品发布后进入待审核状态，却缺少独立的后台作品审核管理页面。目标是保证后端服务、后台管理和用户端的数据一致性，补齐“用户发布 -> 后台审核 -> 用户端展示”的闭环。

## 范围

本次采用“治理型补齐”方案：

1. 新增后台作品管理能力，覆盖作品列表、详情、审核、下架。
2. 增强现有约拍审核和查询页面，补齐新版字段展示与筛选。
3. 修正 `frontend-admin` 的约拍对象、风格、收费方式、作品状态等枚举，统一以后端枚举为事实源。
4. 将现有后台“发布作品”页面更名为“后台代发约拍”，只同步枚举和文案，不重做为新版用户端作品发布流程。

不做的内容：

- 不把约拍和作品合并为一个“内容中心”大模块。
- 不改变用户端已通过内容的公开列表语义。
- 不重构旧 `backend/system-web` 的服务端模板后台。
- 不改变用户发布内容的原始字段，只通过后台审核更新状态和审核理由。

## 后端设计

### 新增后台作品接口

在 `backend/system-wap` 增加后台入口，在 `backend/system-restapi` 和 `backend/system-domain` 增加对应服务能力。

接口：

- `GET /admin/work/list`
  - 用途：后台作品分页。
  - 查询条件：`status`、`nickname`、`mobile`、`city`、`mediaType`、`tagIds`、`page`、`size`。
  - 返回：分页数据，包含作品摘要、封面、作者摘要、标签、状态、计数、发布时间。
- `GET /admin/work/detail`
  - 用途：后台作品详情。
  - 参数：`id`。
  - 返回：作品完整字段、媒体列表、标签、作者信息、审核理由、计数字段。
- `POST /admin/work/audit`
  - 用途：审核通过或不通过。
  - 参数：`id`、`flag`、`reason`。
  - 行为：只更新 `status` 和 `auditReason`。
- `POST /admin/work/offline`
  - 用途：后台下架作品。
  - 参数：`id`、`reason` 可选。
  - 行为：更新 `status=4`，保留原始内容。

服务层新增后台专用方法，避免复用用户端 `pageList` 时误套“只看审核通过”的公开列表条件。后台列表默认展示全部未删除内容，按条件筛选。

### 增强后台约拍接口

保持现有 `/admin/ypat/*` 路径：

- `GET /admin/ypat/list` 增加查询参数：`target`、`patstyle`、`chargeway`、`city`、`workId`。
- `GET /admin/ypat/detail` 返回图片、地区、全国标识、来源作品、风格文本、收费文本、用户摘要和审核原因。

约拍审核继续只更新 `status`、`reason`、`recomflag`，不修改发布内容。

## 一致性规则

1. 枚举值只能在统一枚举工具中定义，页面不再硬编码业务值。
2. 审核动作只更新状态和审核理由，不改用户原始发布字段。
3. 用户端公开列表只展示审核通过内容，后台列表可以查看待审核、通过、驳回、下架状态。
4. 后台展示文本可以适配新版用户端文案，但提交和筛选值必须对齐后端。
5. 约拍来源作品通过 `workId` 关联展示，只做跳转或引用，不自动联动修改作品状态。

- 增加枚举映射、API 参数和分页工具的针对性测试。

静态一致性检查：

- 检索后台菜单，确认“发布作品”已改为“后台代发约拍”。
- 检索作品管理 API 和菜单，确认后台入口完整。

## 风险与控制

- 风险：后台作品列表误用用户端公开列表条件，导致待审核内容不可见。
  - 控制：新增后台专用查询方法，不复用公开列表筛选。
- 风险：枚举文案统一时误改值，影响历史数据。
  - 控制：只改后台前端映射，后端值保持不变，并加静态检查。
- 风险：审核接口修改原始内容。
  - 控制：审核接口只接受 `id`、`flag`、`reason`，服务层只更新状态和理由。
- 风险：页面字段过多导致表格不可读。
  - 控制：列表保留治理关键字段，完整字段放入详情抽屉。
