# YPAT 后端缺口分析

## 概述

当前后端接口基本满足用户端功能需求。以下记录需要前端适配或后端补充的接口差异。

---

## 1. 可通过前端适配解决的问题

### 1.1 响应结构不统一
- **问题**: ResponseApiBody 包装不一致，部分接口返回原始 JSON String，部分返回包装对象
- **影响**: 前端解析逻辑复杂
- **适配方案**: API 适配层统一解析，判断响应是否被 ResponseApiBody 包装

### 1.2 日期格式不统一
- **问题**: 部分接口返回 `yyyy-MM-dd`，部分返回 `yyyy-MM-dd HH:mm:ss`，部分返回时间戳
- **影响**: 前端展示需要多种格式
- **适配方案**: 日期适配器统一转换

### 1.3 图片地址缺少域名
- **问题**: 部分接口返回相对路径，需要拼接 `https://www.91qupaier.com/`
- **影响**: 图片显示
- **适配方案**: 图片 URL 适配器自动补全域名

### 1.4 分页结构
- **问题**: 使用 Spring Data JPA 默认分页结构 (content, totalElements, number, size)
- **影响**: 前端需要适配 Spring 分页格式
- **适配方案**: 分页适配器转换为统一格式

### 1.5 枚举字段为代码而非文字
- **问题**: 如 status="shtg", gender="1", profess="0"
- **影响**: 前端需要维护枚举映射
- **适配方案**: 前端常量文件维护所有枚举映射

---

## 2. 后端接口可能缺失的功能

### 2.1 手机号登录
- **现状**: 旧前端仅支持微信/百度 OAuth，无独立手机号登录接口
- **影响**: H5 端需要手机号+验证码登录
- **推荐方案**: 前端 H5 通过微信内置浏览器授权登录，或使用现有 mobile 字段做简化登录
- **兼容方案**: H5 引导用户使用微信小程序
- **回滚方案**: H5 暂不提供独立登录，仅作为分享落地页

### 2.2 图片直传
- **现状**: 图片通过后端 FastDFS 上传，无前端直传 OSS 方案
- **影响**: 上传速度、并发能力
- **推荐方案**: 保持现有后端上传方式
- **兼容方案**: 前端做图片压缩后上传
- **回滚方案**: 无需回滚

### 2.3 搜索接口
- **现状**: 仅支持按字段筛选，无全文搜索
- **影响**: 用户搜索体验
- **推荐方案**: 前端使用现有筛选接口组合实现搜索效果
- **兼容方案**: 用 describ 字段模糊匹配（需确认后端是否支持）
- **回滚方案**: 仅提供筛选，不提供搜索

### 2.4 取消收藏
- **现状**: 只有 /my/ypat/sc/add (添加收藏)，未见取消收藏接口
- **影响**: 用户无法取消收藏
- **推荐方案**: 确认 sc/add 是否为 toggle 操作（旧前端调用同一接口）
- **兼容方案**: 前端调用 sc/add 实现 toggle
- **回滚方案**: 仅支持添加收藏

### 2.5 意见反馈提交
- **现状**: 未发现专门的反馈提交接口
- **影响**: 用户无法提交反馈
- **推荐方案**: 使用客服消息或邮件方式
- **兼容方案**: 前端展示联系方式引导反馈
- **回滚方案**: P2 功能，可延后

### 2.6 删除约拍
- **现状**: 有 DELETE /service/ypat/del 但前端入口不明确
- **影响**: 用户可能需要删除自己的约拍
- **推荐方案**: 在"我的发布"中增加删除入口
- **兼容方案**: 仅允许删除草稿状态的约拍
- **回滚方案**: 暂不开放删除

## 3. 需要确认的接口行为

| 接口 | 疑问 | 影响 |
|------|------|------|
| /my/ypat/sc/add | 重复调用是 toggle 还是报错？ | 收藏/取消收藏逻辑 |
| /ypat/submit | 草稿保存和正式提交是否同一接口？ | 发布流程设计 |
| /user/upd | 头像上传是否单独接口？ | 编辑资料流程 |
| /oauth/ocr | 支持正反面分别识别吗？ | OCR 流程设计 |
| /order/create | 返回的支付参数格式？ | 微信支付对接 |
| /wxpay/notify | 前端如何确认支付成功？ | 支付结果页 |

---

## 4. 前端不需要修改后端即可实现的方案

所有已识别的差异均可通过前端适配层解决，当前不需要修改后端接口。

关键适配策略：
1. **API 适配层**: 统一响应格式解析
2. **DTO 转换层**: 枚举代码→文字映射
3. **字段标准化**: 日期格式统一
4. **空值兼容**: null/undefined/空字符串统一处理
5. **图片地址处理**: 自动补全域名前缀
6. **分页结构适配**: Spring Data → 统一分页格式
7. **错误码适配**: 后端错误码→前端用户提示
