# YPAT API Contract Specifications

This directory is the **source of truth for the HTTP API surface** of YPAT.
Every breaking change to the public API must land here before code lands.

## Files

| File | Status | Owner |
|---|---|---|
| `openapi-v1-baseline.yaml` | Placeholder (PR-00) | backend team |
| `openapi-v2-candidate.yaml` | Will be generated by PR-04 | backend team |
| `*.golden.json` | Will be recorded by WP-01 | QA + backend |

## Workflow

1. **Phase 0 (current)** — Only `openapi-v1-baseline.yaml` exists, intentionally
   near-empty. Spectral runs against it in CI to keep the structure honest.
2. **Phase 1 (PR-04)** — `backend-v2` lands with `springdoc-openapi:2.x` and
   the first real baseline is generated from controllers, then reviewed and
   locked.
3. **Phase 1+ (every PR)** — Any new endpoint, parameter, response shape, or
   error code MUST be reflected in the spec on the same PR.

## Linting

CI runs [`stoplightio/spectral-action`](../../.github/workflows/openapi-lint.yml)
against every file in `spec/` using the rule set in [`.spectral.yaml`](../../.spectral.yaml).

| Rule class | What it catches |
|---|---|
| Metadata | Missing `info.title` / `info.version` / `servers` |
| Operation | Missing `operationId`, `summary`, `tags` |
| Parameters & responses | Missing schemas, undocumented path params |
| Uniqueness | Duplicate `operationId` across paths |

## Authoring rules

- **One file per major version**. Don't merge v1 + v2 specs into one file.
- **Never hand-edit generated files.** If you must change generated content,
  change the generator or the controller and regenerate.
- **Breaking changes** require a row in `docs/architecture/breaking-changes.md`
  (created by PR-04 or later) and sign-off from the upgrade lead.
