# 一、核心任务

当前仓库中的 `frontend` 目录虽然已经初始化为 UniApp、Vue 3、TypeScript 和 Vite 项目，但基本属于空白脚手架。

你的任务不是简单修改几个页面，而是：

1. 全面分析当前仓库中的后端服务、Controller、DTO、实体、配置、接口和业务流程。
2. 梳理原项目全部用户端功能。
3. 建立完整的功能迁移矩阵。
4. 使用 UniApp、Vue 3、TypeScript 重新实现用户端。
5. 保留现有后端业务能力和接口契约。
6. 使用全新的高质量 UI 设计。
7. 同时支持：

   * 微信小程序；
   * H5；
   * App；
8. 优先保证微信小程序可正式审核和上线。
9. 达到企业级项目在架构、代码质量、测试、性能、安全、兼容性和发布方面的标准。
10. 最终必须交付可运行、可测试、可构建、可部署、可上线的完整项目。

不要只输出分析文档或实施计划，必须继续完成实际代码开发。

# 三、强制执行原则

## 3.1 不允许凭空实现

任何业务页面和接口都必须有明确来源：

* 后端 Controller；
* DTO、VO、Entity；
* README；
* 数据库结构；
* 原有静态资源；
* 配置文件；
* 业务代码；
* 已有接口调用逻辑。

无法确认的功能必须记录在：

```text
docs/migration/open-questions.md
```

不要直接虚构字段、接口地址、支付参数或认证流程。

## 3.2 保持后端兼容

默认不得修改后端接口。

前端应通过以下方式兼容历史接口：

* API 适配层；
* DTO 转换层；
* 字段标准化；
* 空值兼容；
* 枚举映射；
* 时间格式转换；
* 图片地址处理；
* 分页结构适配；
* 错误码适配。

只有在后端接口确实无法满足业务时，才允许提交最小后端修改，并在以下文档中说明：

```text
docs/migration/backend-gap-analysis.md
```

文档必须记录：

* 原接口问题；
* 影响范围；
* 推荐修改方案；
* 兼容方案；
* 回滚方案。

## 3.3 不允许使用假接口完成交付

开发阶段允许使用 Mock，但必须满足：

* Mock 与真实接口类型一致；
* Mock 只能在开发环境启用；
* 生产环境不得访问 Mock；
* Mock 数据必须集中管理；
* 页面不得直接写死业务数据；
* 接入真实接口后必须完成联调。

## 3.4 不允许只生成静态页面

所有页面必须实现真实交互状态，包括：

* 加载中；
* 首次加载；
* 下拉刷新；
* 上拉加载；
* 空状态；
* 错误状态；
* 网络异常；
* 登录失效；
* 无权限；
* 提交中；
* 提交成功；
* 提交失败；
* 重复点击；
* 数据回显；
* 表单校验；
* 返回恢复；
* 页面缓存。

## 3.5 不允许遗留占位代码

正式交付代码中不得存在：

```text
TODO
FIXME
临时处理
后续实现
模拟提交成功
假支付
假登录
静态 Token
硬编码用户信息
硬编码生产域名
```

用户可见的按钮、弹窗、提示、错误信息和表单校验文案全部使用中文。

## 5.2 架构职责

必须做到：

* 页面组件不直接调用 `uni.request`；
* 页面组件不直接拼接接口地址；
* 页面组件不直接解析历史后端异常结构；
* 页面组件不直接操作原始 Token；
* 所有接口集中在 `api/modules`；
* 所有后端字段转换集中在 `api/adapters`；
* 全局状态集中在 Pinia；
* 可复用业务逻辑进入 composables；
* 可复用 UI 进入 components；
* 枚举、状态和常量集中管理；
* 环境变量集中管理；
* 不同平台差异集中封装；
* 不允许在业务代码中到处分散条件编译。
