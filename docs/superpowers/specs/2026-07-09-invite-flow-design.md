# 邀请链路完整闭环设计

日期：2026-07-09
工作区：`/Users/lizhenwei/workspace/vueworkspace/ypat-workspace`
目标端：`frontend`（新版 UniApp 小程序前端）、`frontend-admin`（管理后台）、`backend`（Spring Cloud 后端）

## 背景

新版 `frontend`（小程序前端）已经存在邀请基础能力：

- `frontend/src/pages-sub/user/invite.vue`：我的邀请页。
- `frontend/src/pages-sub/user/invite-records.vue`：邀请记录页。
- `frontend/src/pages-sub/content/invite-landing.vue`：邀请落地页。
- `frontend/src/services/invite-context.ts`：保存分享进入时的 `inviteCode`（邀请码）和 `source`（来源）。
- 登录页会把邀请上下文传给 `/user/login`。

后端已经存在邀请关系与基础接口：

- `InviteCodeCodec`（邀请码编解码）。
- `InviteRelation`（邀请关系实体）和 `InviteService`（邀请服务）。
- `/invite/my-info`、`/invite/rule`、`/invite/records` 小程序侧接口。

当前缺口：

1. 邀请规则仍主要依赖后端常量，无法由管理后台配置。
2. 管理后台没有邀请配置页面和邀请数据展示页面。
3. 注册链路中发奖发生在关系写入前，幂等关系虽然能兜底，但边界上应以“关系创建成功”为发奖前置条件。
4. 小程序“我的邀请页”和“邀请落地页”视觉偏基础，没有形成明确的邀请转化页。

本设计采用完整闭环方案：补齐后台配置、后台数据展示、后端配置服务、后端发奖幂等、小程序邀请活动页和分享落地页。

## 目标

- 后台新增“邀请运营”模块，包含“邀请配置”和“邀请记录”两个页面。
- 邀请基础配置支持后台维护：
  - 邀请功能启用/停用。
  - 每成功邀请 1 人奖励多少拍拍豆。
  - 规则说明文案。
  - 分享标题。
  - 邀请落地页主文案。
- 小程序 `/invite/rule`（邀请规则接口）从后台配置读取，默认奖励仍为 3 拍拍豆。
- 我的邀请页升级为更有活动感的邀请页，保持现有绿色主题，奖励数字使用少量金色/橙色强调。
- 微信好友点击分享卡片后先进入邀请落地页，再登录/注册并绑定邀请关系。
- 后端发奖逻辑调整为关系创建成功后才发放奖励，避免重复发奖。
- 邀请记录可在小程序个人侧查看，也可在管理后台按筛选条件分页查看。
- 不修改 `91pai-master`（旧小程序）。

## 非目标

- 不做多期活动、活动开始/结束时间、奖励阶梯、海报生成、二维码海报、渠道归因报表。
- 不做邀请奖励撤销、人工补发、人工改绑。
- 不引入通用营销活动系统。
- 不改变充值、会员、签到、发布约拍等其他拍拍豆链路。
- 不下载依赖，不做全量构建；只做邀请相关轻量测试或源代码级校验。

## 方案选择

采用方案 A：完整闭环，范围可控。

理由：

1. 用户要求同时检查并补齐小程序邀请链路、管理后台数据展示和后端服务。
2. 后端已有邀请关系、邀请码和基础接口，补后台配置与后台记录页能复用现有结构。
3. 把配置能力做成基础配置即可满足当前运营诉求，不需要扩大到活动系统。
4. 发奖幂等是当前链路的关键稳定性问题，适合随本次闭环一起收紧。

未采用方案：

1. 最小补齐：只做配置不做记录页，会漏掉“后台数据展示”的明确要求。
2. 活动化方案：支持多期活动和渠道统计，但会扩大数据模型和验收范围。

## 总体架构

后端新增邀请配置领域能力。`backend/system-domain` 负责配置读取、配置保存、邀请关系列表查询和注册发奖事务。`backend/system-restapi` 暴露服务间接口。`backend/system-wap` 暴露小程序接口和后台管理接口。

管理后台 `frontend-admin` 新增“邀请运营”菜单分组，包含配置表单页和邀请记录表格页。

小程序 `frontend` 保留现有页面路径，重点升级页面结构与视觉：

- `pages-sub/user/invite`：登录用户的“我的邀请”主页面。
- `pages-sub/content/invite-landing`：被邀请用户从微信分享进入后的主落地页。
- `pages-sub/user/invite-records`：个人邀请记录页，保留并适度优化空态和奖励展示。

## 数据模型

### `t_invite_config`

邀请配置表保存单条全局配置。

字段：

| 字段 | 类型 | 默认值 | 含义 |
| --- | --- | --- | --- |
| `id` | `BIGINT` | 自增 | 主键 |
| `enabled` | `VARCHAR(1)` | `1` | 是否启用：`1` 启用，`0` 停用 |
| `reward_ppd` | `INT` | `3` | 每成功邀请 1 人奖励拍拍豆数 |
| `rule_text` | `VARCHAR(300)` | 默认规则文案 | 规则说明文案 |
| `share_title` | `VARCHAR(120)` | 默认分享标题 | 微信分享卡片标题 |
| `landing_title` | `VARCHAR(160)` | 默认落地页主文案 | 邀请落地页主文案 |
| `created_at` | `DATETIME` | 当前时间 | 创建时间 |
| `updated_at` | `DATETIME` | 当前时间 | 更新时间 |

校验：

- `enabled` 只能是 `0` 或 `1`。
- `reward_ppd` 必须大于等于 0，建议后台限制上限为 999。
- `rule_text`、`share_title`、`landing_title` 不能为空且限制长度，避免小程序文案溢出。

默认值：

- `enabled = "1"`。
- `reward_ppd = 3`。
- `rule_text = "好友通过你的邀请码注册后，自动到账 3 拍拍豆。"`。
- `share_title = "好友邀请你加入爱去拍，找摄影师、找模特更方便"`。
- `landing_title = "我正在使用爱去拍，找摄影师、找模特特别方便，推荐你也来体验。"`。

### `t_invite_relation`

继续使用现有邀请关系表。

本次不破坏已有字段和唯一约束：

- `uk_invitee_userid(invitee_userid)`：同一个被邀请用户只能绑定一次。
- `idx_inviter_userid(inviter_userid)`：按邀请人查询。

可选增强：

- 查询后台记录时按 `inviter_userid`、`invitee_userid`、`invite_code`、`source` 过滤。
- 列表返回邀请人和被邀请人的用户快照。若一次性补双边用户信息会产生明显 N+1 查询，第一版优先补被邀请人昵称和脱敏手机号，邀请人保持 ID 展示。

### 拍拍豆流水

邀请奖励继续复用现有 `t_record`：

- `type = RecordType.FRI`（好友邀请）。
- `ppd = 当前邀请配置 reward_ppd`。
- `userid = 邀请人 ID`。

关键调整：只有邀请关系新建成功后才写拍豆流水和增加邀请人 `ppd`。

### `GET /invite/my-info`

登录后获取当前用户邀请概览。

返回字段保持兼容：

- `inviteCode`: 当前用户邀请码。
- `totalInvited`: 成功邀请人数。
- `totalReward`: 累计邀请奖励。
- `rewardPpd`: 当前配置奖励数。

新增可选字段：

- `enabled`: 是否启用邀请功能。

### `GET /invite/records`

保持现有个人邀请记录分页接口。后端从 `SecurityContext`（安全上下文）取当前用户 ID，不接受前端伪造邀请人 ID。

## 后端事务设计

注册链路中的邀请处理调整为：

1. 新用户注册成功并生成 `user.id`。
2. 读取 `InviteConfig`（邀请配置）。
3. 如果邀请功能停用，跳过邀请绑定和发奖。
4. 优先解析 `inviteCode`，退化兼容 `recmobile`。
5. 无有效邀请人、自我邀请时跳过，不影响注册。
6. 调用 `InviteService.bindRelationIfAbsent`（不存在则绑定关系）：
   - 邀请人和被邀请人不能为空。
   - 邀请人和被邀请人不能相同。
   - 如果被邀请人已有关系，返回未创建。
   - 如果创建成功，返回创建的关系。
   - 并发唯一约束冲突按未创建处理。
7. 只有关系创建成功且 `reward_ppd > 0` 时：
   - 增加邀请人 `ppd`。
   - 写入 `RecordType.FRI`（好友邀请）拍拍豆流水。
   - 关系上的 `rewardPpd` 记录当次实际奖励数。

这样保证同一个被邀请用户即使重复触发注册请求或并发请求，也不会重复给邀请人加豆。

### 我的邀请页

页面目标：像活动页一样明确告诉用户“邀请好友可以获得奖励”，同时保持爱去拍当前绿色主题。

结构：

   - 副标题来自 `ruleText`。
   - 使用绿色渐变背景，加入摄影/同伴/奖励的轻量装饰元素。
3. 奖励卡片：
   - 文案：“每成功邀请 1 位好友”。
   - 大号数字：“+X 拍拍豆”。
   - 使用金色/橙色强调数字，不改变整体绿色主题。
4. 数据卡片：
   - 成功邀请人数。
   - 累计奖励。

6. 规则说明：

## 错误处理

### 配置缺失

后端返回默认配置。后台配置页首次打开时展示默认值，管理员保存后落库。

### 邀请功能停用

小程序：

- 我的邀请页禁用分享/复制按钮。
- 邀请落地页降级为平台介绍页，不展示奖励承诺。

后端：

- `/invite/rule` 返回 `enabled=false`。
- 注册链路不绑定邀请关系、不发奖励。

### 邀请码无效

注册正常完成，不绑定邀请关系、不发奖励，不向用户展示阻断错误。

### 自我邀请

后端拒绝绑定，不发奖励。前端不额外提示，避免干扰注册。

### 重复绑定

关系已存在时不创建新关系、不发奖励、不写重复流水。

### 接口失败

小程序页面使用默认文案和空态兜底；后台页面使用 Element Plus（后台 UI 组件库）消息提示。

## 验收标准

1. 从 `origin/main` 创建独立 worktree（独立工作区）进行开发，不污染当前 `dev6` 主工作区的未提交改动。
2. `frontend`（新版小程序前端）我的页仍能进入“好友邀请”。
3. 我的邀请页展示后台配置的奖励数、规则文案、成功邀请人数、累计奖励、邀请码。
4. 我的邀请页在微信小程序端可分享给好友，分享路径进入邀请落地页并携带 `inviteCode` 和 `source=share`。
5. 被邀请用户点击分享卡片后进入邀请落地页，而不是直接进入登录页或首页。
6. 邀请落地页点击“马上体验”后进入登录页，并保留邀请上下文。
7. 被邀请用户注册成功后，后端按邀请码绑定邀请关系。
8. 只有邀请关系新建成功时才给邀请人发放配置中的拍拍豆。
9. 同一个被邀请用户重复触发注册或并发触发时，不重复发放邀请奖励。
10. 邀请功能被后台停用时，小程序不展示奖励承诺，后端不绑定新邀请关系、不发奖励。
11. 管理后台存在“邀请运营”模块。
12. 管理后台“邀请配置”页可查看和保存基础配置。
13. 管理后台“邀请记录”页可分页查看邀请关系，并支持按邀请人 ID、被邀请人 ID、邀请码、来源筛选。
14. `/invite/rule` 返回后台配置，配置缺失时返回安全默认值。
15. 不修改 `91pai-master`（旧小程序）。

## 实施顺序建议

1. 从 `origin/main` 创建独立 worktree。
2. 后端补邀请配置对象、服务、服务间接口、后台接口。
3. 后端调整注册发奖事务，补幂等测试。
4. 管理后台补 API、菜单、路由映射、配置页、记录页。
5. 小程序补 API 类型字段，优化我的邀请页和邀请落地页。
6. 运行邀请相关轻量测试和源代码检查。
7. 汇总改动与验证证据。
